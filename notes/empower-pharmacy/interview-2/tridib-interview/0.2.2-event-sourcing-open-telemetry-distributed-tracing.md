**Event Sourcing — What It Actually Means in Code** 
**Without event sourcing** (the traditional way), you just update a row:
```java
// Traditional: UPDATE the prescription row directly
prescription.setStatus("COMPOUNDING");
prescriptionRepository.save(prescription);
// Problem: the old status is gone forever. No trace of who changed it or why.
```

  **With event sourcing**, you never update. You only append events:
```java
// Event sourcing: APPEND a new event describing what happened
PrescriptionEvent event = new PrescriptionEvent();
event.setEventId(UUID.randomUUID());
event.setPrescriptionId("RX-1234");
event.setEventType("STATUS_CHANGED");
event.setOldStatus("VALIDATED");
event.setNewStatus("COMPOUNDING");
event.setChangedBy("pharmacist-jane");
event.setReason("QA passed, ingredients available");
event.setCorrelationId("trace-5678");  // same ID across all services
event.setTimestamp(Instant.now());
// Append-only — this is an INSERT, never an UPDATE or DELETE
eventStore.append(event);
```

**The event store table** looks like this:
```sql
CREATE TABLE prescription_events (
    event_id     UUID PRIMARY KEY,
    prescription_id  VARCHAR NOT NULL,
    event_type   VARCHAR NOT NULL,    -- STATUS_CHANGED, CREATED, INGREDIENT_RESERVED, etc.
    old_value    JSONB,
    new_value    JSONB,
    changed_by   VARCHAR NOT NULL,    -- who did this
    reason       VARCHAR,             -- why they did it
    correlation_id VARCHAR NOT NULL,  -- links across services
    created_at   TIMESTAMP NOT NULL
    -- NO update_at column. Rows are never updated.
);
```

**To get the current state of a prescription**, you replay its events:
```java
List<PrescriptionEvent> history = eventStore
    .findByPrescriptionId("RX-1234")
    .orderBy(createdAt);
// Replay events to build current state
Prescription current = new Prescription();
for (PrescriptionEvent event : history) {
    current.apply(event);  // each event modifies the in-memory object
}
// current.getStatus() → "COMPOUNDING"
```

**For HIPAA audit**, you just query the event store directly — the audit trail _is_ the data:
```sql
-- "Show me everything that happened to prescription RX-1234"
SELECT event_type, old_value, new_value, changed_by, reason, created_at
FROM prescription_events
WHERE prescription_id = 'RX-1234'
ORDER BY created_at;
-- Result:
-- 2026-05-01 09:00  CREATED           null → {drug: Lisinopril}  dr-smith    "New Rx"
-- 2026-05-01 09:05  STATUS_CHANGED    CREATED → VALIDATED        system      "Auto-validated"
-- 2026-05-01 09:10  INVENTORY_RESERVED null → {resId: res-5678}  system      "Ingredients reserved"
-- 2026-05-01 09:15  STATUS_CHANGED    VALIDATED → COMPOUNDING    pharmacist-jane  "QA passed"
```

No separate audit table needed. The event store _is_ the audit log.

**OpenTelemetry Distributed Tracing — What It Actually Does** 
**The problem:** A prescription request touches 5 services. When something is slow, which service caused it?
**The solution:** Every request gets a `traceId` that follows it everywhere:
```java hl:4,7-19
// In the Order Service — the request starts here
@RestController
public class OrderController {
	@Autowired private Tracer tracer;
    @PostMapping("/orders")
    public Order createOrder(@RequestBody OrderRequest req) {
        // OpenTelemetry automatically creates a traceId for this request
        // traceId: "abc-123" — this same ID will appear in EVERY service
        Span span = tracer.spanBuilder("createOrder").startSpan();
        // Call the Inventory Service via WebClient
        // OpenTelemetry AUTOMATICALLY adds the traceId to the HTTP header:
        //   Header: traceparent: 00-abc123-def456-01
        webClient.post()
            .uri("http://inventory-service/reserve")
            .body(req.getItems())
            .retrieve();
        // The Inventory Service receives this header,
        // so its logs and spans also carry traceId "abc-123"
        span.end();
        return order;
    }
}
@Configuration
public class TracingConfig {
    @Bean
    public Tracer tracer() {
        // OpenTelemetry gives you a Tracer tied to your service name
        return GlobalOpenTelemetry.getTracer("order-service");
    }
}
```

  **What you see in Azure Monitor / Jaeger:**
```
traceId: abc-123
│
├── Order Service: createOrder ──────────────── 250ms total
│   ├── Validate prescription ──────────────── 15ms
│   ├── Call Inventory Service ─────────────── 180ms  ← slow!
│   │   ├── Check ingredient availability ──── 5ms
│   │   ├── Reserve stock (DB write) ───────── 170ms  ← HERE'S THE BOTTLENECK
│   │   └── Send reply ────────────────────── 5ms
│   └── Publish event to Service Bus ───────── 8ms
```

**In the logs**, every service includes the same `traceId`:
```
// Order Service log:
{"traceId":"abc-123", "service":"order", "msg":"Creating order RX-1234"}

// Inventory Service log:
{"traceId":"abc-123", "service":"inventory", "msg":"Reserving 500ml Lisinopril"}

// Payment Service log:
{"traceId":"abc-123", "service":"payment", "msg":"Charging insurance $45.00"}
```

**To find all logs for one prescription across all 5 services:**
```sql
-- One query in Azure Log Analytics:
traces | where customDimensions.traceId == "abc-123"
```

That single query returns logs from Order, Inventory, Payment, Compounding, and Shipping — all stitched together by the `traceId` that OpenTelemetry propagated automatically.
**The magic:** You add one dependency to each service and OpenTelemetry does the rest:
```xml hl:3-4
<!-- pom.xml — this is all you add -->
<dependency>
    <groupId>io.opentelemetry.javaagent</groupId>
    <artifactId>opentelemetry-javaagent</artifactId>
</dependency>
```

It automatically instruments HTTP calls, database queries, and message broker operations. No manual code needed for 90% of tracing.
