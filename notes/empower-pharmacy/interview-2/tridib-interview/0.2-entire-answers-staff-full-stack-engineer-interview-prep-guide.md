## I. Architecture & System Design (Q1–Q6)

### Q1: "Design a system that handles prescription order processing at scale — from intake through fulfillment — with traceability at every step."

I'd break this into five bounded contexts: **Order Intake**, **Prescription Validation**, **Inventory**, **Manufacturing/Compounding**, and **Fulfillment/Shipping**. Each runs as an independent Spring Boot Reactive microservice on AKS.

The frontend is a Next.js application — pharmacy staff use it for order management dashboards, patient lookup, and status tracking. It talks to a **GraphQL gateway** that acts as the orchestration layer, aggregating data from multiple backend services into a single query.

For the flow: a prescription comes in via the Order Intake service, which validates the payload and publishes an `OrderReceived` event to **Azure Service Bus**. The Prescription Validation service subscribes, checks drug interactions, verifies provider credentials, and publishes `OrderValidated` or `OrderRejected`. The Inventory service reserves stock, the Compounding service queues the manufacturing work order, and the Fulfillment service handles shipping label generation and carrier integration.

For traceability — the critical part in a regulated pharmacy — I'd use an **event sourcing** pattern. Every state change is an immutable event stored in an append-only event store. This gives us a complete audit trail: who changed what, when, and why. Each event includes a correlation ID that threads through all services, so we can reconstruct the full lifecycle of any prescription.

Observability: OpenTelemetry for distributed tracing across all services, structured logging shipped to Azure Monitor, and Prometheus metrics scraped by Grafana dashboards on AKS.

### Q2: "Walk me through how you'd architect a service that coordinates manufacturing workflows across multiple facilities with real-time status tracking."

Compounding pharmacy manufacturing is a **stateful, multi-step workflow** — ingredients are weighed, mixed, tested for quality, packaged, and labeled. Multiple facilities add coordination complexity.

I'd model this as an **orchestration service** using a durable workflow engine — either Temporal or a custom Spring Boot orchestrator backed by a state machine. Each manufacturing order is a workflow instance with defined states: `QUEUED → ASSIGNED_TO_FACILITY → COMPOUNDING → QA_TESTING → PACKAGING → READY_TO_SHIP`.

**Facility assignment:** When an order arrives, the orchestrator evaluates which facility should handle it based on ingredient availability, current workload, equipment capabilities, and geographic proximity to the shipping destination. This is a rules engine call — not hardcoded logic — so operations can update routing rules without code changes.

**Real-time status tracking:** Each facility has local workstation apps (tablet-based React/Next.js apps) where technicians scan barcodes to advance the workflow state. Each scan publishes a `WorkflowStepCompleted` event to Azure Service Bus. The orchestrator updates the workflow state and publishes a `StatusChanged` event. The frontend subscribes via GraphQL subscriptions (WebSocket) for live updates.

**Cross-facility coordination:** If Facility A runs out of an ingredient mid-compounding, the orchestrator triggers a transfer request to Facility B, or re-routes the order entirely — a compensating action in the saga.

**Data model:** Each facility runs its own local services for speed and resilience (if the network between facilities goes down, local manufacturing continues). State syncs to the central orchestrator via Service Bus with guaranteed delivery. The central service is the source of truth for cross-facility views; local services are the source of truth for in-facility operations.

### Q3: "How would you design a backend that handles high-throughput order processing while maintaining HIPAA-compliant audit trails?"

The tension is that audit logging adds overhead to every operation, and at high throughput, that compounds. The solution: make audit logging **asynchronous and non-blocking** so it doesn't sit in the critical path.

**Order processing path (optimized for throughput):** Spring Boot Reactive services process orders using non-blocking I/O. Each service uses R2DBC for database access and WebClient for inter-service calls. The pipeline — validate, check inventory, reserve, confirm — runs as a reactive chain with backpressure control. Target: 1,000+ orders per minute.

**Audit trail path (optimized for completeness):** At each step, the service emits an audit event to an in-memory buffer (Reactor's `Sinks.many().multicast()`). A background subscriber drains this buffer and publishes events to a dedicated Azure Service Bus topic. A separate **Audit Service** consumes these and writes to an append-only store (Cosmos DB with immutability enabled).

This decoupling means order processing latency is unaffected by audit logging speed. If the Audit Service is temporarily down, events queue in Service Bus (14-day retention) — no audit data is lost.

**HIPAA specifics:** Every audit event includes: user ID, timestamp (UTC), action performed, resource type and ID, fields accessed or modified (but NOT the PHI values themselves), source IP, and a correlation ID. Encryption in transit (TLS 1.3) and at rest (Azure SSE with customer-managed keys). Access to audit data restricted to compliance roles via Azure RBAC.

**Retention:** HIPAA requires 6 years. Tiered storage: hot (90 days in Cosmos DB), warm (90 days–2 years in Table Storage), cold (2–6 years in immutable blob storage).

### Q4: "How do you decide service boundaries? Walk me through a real example."

I align service boundaries with **business domain boundaries** using Domain-Driven Design. The question I ask: "If this capability were owned by a different team tomorrow, what data and logic would they need to operate independently?"

**Example:** I worked on a system that started as a monolith handling both customer orders and warehouse operations. The pain point: deploying a change to the order flow required retesting warehouse logic — and warehouse had a completely different release cadence.

We ran an event-storming workshop. Two bounded contexts emerged: **Order Management** (customer-facing, needed fast iteration) and **Warehouse Operations** (operational, needed stability). The shared concept was "order" — but it means different things in each context. In Order Management, it's a customer's intent to purchase. In Warehouse, it's a pick list with bin locations.

We split into two services with their own databases (database-per-service pattern). They communicated via events: Order Management published `OrderConfirmed`, Warehouse subscribed and created its own internal representation. Order Management shipped 3x a week; Warehouse shipped biweekly.

The anti-pattern I watch for: splitting by technical layer (a "user service," a "notification service") instead of by business capability.

### Q5: "How do you handle distributed transactions across services?"

I avoid distributed transactions wherever possible — two-phase commit doesn't scale. Instead, I use the **Saga pattern** with eventual consistency.

For a pharmacy order flow, I'd use an **orchestration-based saga**. The business process has a clear sequence (validate → reserve inventory → compound → ship), and if any step fails, we need specific compensating actions. An orchestrator service owns this workflow.

If the Compounding step fails (ingredient out of stock), the orchestrator triggers compensating actions: release the inventory reservation, notify the prescriber, update the order status to "on hold." Each step is idempotent — if a message is delivered twice, the service detects the duplicate via a unique saga ID and skips it.

Implementation: **Azure Service Bus sessions** for ordered message processing within a saga, and dead-letter queues for messages that fail after retries. The orchestrator persists its state so it can recover from crashes.

The alternative — choreography — works for simpler flows. But for a regulated pharmacy workflow with 5+ steps and complex failure modes, orchestration gives better visibility into where a process is stuck.

### Q6: "How do you manage data ownership when multiple services need the same data?"

Each service owns its data — no shared databases. When another service needs that data, I use one of three patterns:

- **Event-carried state transfer:** The owning service publishes events with data payloads. Consuming services maintain their own read-optimized local copies. Example: the Patient service publishes `PatientUpdated` events; the Order service keeps a local projection of patient name, allergies, and insurance info.
- **API query at runtime:** When data is too large to replicate or changes too frequently, the consuming service queries the owning service's API directly. Add caching (Redis) and circuit breakers.
- **GraphQL aggregation:** The GraphQL layer resolves data from multiple services in a single query using DataLoader batching.

The key discipline: one service is always the **source of truth** for any given entity. If two services seem to co-own data, the boundaries are wrong.

## II. Core Stack Deep Dive (Q7–Q22)

### Q7: "Why Spring WebFlux over traditional Spring MVC? When would you not use reactive?"

Spring MVC uses a **thread-per-request** model — each request occupies a thread for its entire lifecycle, including I/O waits. Under high concurrency, you exhaust the thread pool.

WebFlux uses an **event-loop** model with non-blocking I/O. A small number of threads (one per CPU core) handle many concurrent requests by releasing the thread while waiting for I/O. For a pharmacy system processing thousands of concurrent order status checks, prescription validations, and inventory queries — most I/O-bound — this means dramatically higher throughput on the same hardware.

When I would **not** use reactive:

- **CPU-bound workloads** — reactive doesn't help because you're not waiting on I/O.
- **Simple CRUD with low concurrency** — MVC is simpler and the team is more productive.
- **Legacy dependencies** — if most downstream calls are blocking JDBC or SOAP, you're wrapping everything in `Schedulers.boundedElastic()` anyway, negating the benefit.

For Empower's architecture, reactive makes sense for orchestration and data-access-heavy services. For a simple internal admin tool, I'd use MVC.

### Q8: "Explain backpressure in Project Reactor."

Backpressure is the mechanism by which a **consumer tells a producer to slow down** when it can't keep up. Without it, a fast producer overwhelms a slow consumer, causing OOM errors or dropped data.

In Project Reactor, this follows the Reactive Streams spec. When a subscriber subscribes to a `Flux`, it calls `request(n)` to say "give me n items." The publisher only emits that many. Once processed, the subscriber requests more. This pull-based model prevents unbounded buffering.

Practical example: a `Flux` streaming prescription orders from Azure Service Bus. The downstream validation service processes each order (~50ms for a drug interaction check). If orders arrive faster, backpressure kicks in — the Flux buffers a bounded number and signals the consumer to stop pulling.

Operators that affect backpressure:
- `onBackpressureBuffer(maxSize)` — buffers up to maxSize, then errors or drops
- `onBackpressureDrop()` — drops items the consumer can't handle
- `limitRate(n)` — prefetches n items at a time from upstream

### Q9: "How do you handle blocking calls inside a reactive pipeline?"

Never run blocking code on event-loop threads. Offload to a **dedicated thread pool**:

```java
Mono.fromCallable(() -> legacyJdbcRepository.findById(id))
    .subscribeOn(Schedulers.boundedElastic())
    .flatMap(result -> processReactively(result));
```

`fromCallable` wraps the blocking call; `subscribeOn(boundedElastic)` ensures it runs on a separate, bounded thread pool (capped at 10x CPU cores). The rest of the pipeline stays on event-loop threads.

For database access, push to adopt **R2DBC** — a truly non-blocking driver for PostgreSQL, MySQL, SQL Server. For legacy SOAP or REST APIs, the boundedElastic approach is the pragmatic choice.

Key mistake to avoid: using `block()` anywhere in a reactive pipeline. That defeats the purpose and throws `IllegalStateException` on non-blocking threads.

### Q10: "Walk me through a reactive pipeline that processes prescription orders, validates, checks inventory, and publishes events — all non-blocking."

1. **Receive:** Message arrives from Azure Service Bus. The reactive receiver gives us a `Flux<ServiceBusReceivedMessage>`.

2. **Deserialize & Validate:** `flatMap` to parse JSON into a `PrescriptionOrder` DTO, then schema validation. Invalid payloads routed to dead-letter via `onErrorResume`.

3. **Drug Interaction Check:** `flatMap` into a non-blocking WebClient call to the Drug Interaction service. Returns `Mono<InteractionResult>`. Critical interaction → short-circuit with `OrderRejected`.

4. **Inventory Check:** `flatMap` into WebClient call to Inventory. Uses `Mono.zip` to check all ingredient availability in parallel.

5. **Publish Event:** If all pass, publish `OrderValidated` to Service Bus via the reactive sender.

6. **Error handling:** `retryWhen(Retry.backoff(3, Duration.ofSeconds(1)))` for transient failures. `onErrorResume` for permanent failures — log, dead-letter, move on. Each step logs with a correlation ID.

The entire pipeline is non-blocking. A handful of Netty event-loop threads handle thousands of concurrent prescriptions.

### Q11: "How do you test reactive code?"

**StepVerifier** is the core tool:

```java
StepVerifier.create(prescriptionService.validate(order))
    .expectNext(ValidationResult.APPROVED)
    .verifyComplete();
```

Error scenarios: `.expectError(DrugInteractionException.class)`. Backpressure: `.thenRequest(5).expectNextCount(5)`.

For time-dependent operations, StepVerifier's **virtual time** mode: `StepVerifier.withVirtualTime(() -> service.retryableCall())` fast-forwards through retry delays.

Integration tests: **Testcontainers** for a real Service Bus emulator and database, with `WebTestClient` for end-to-end reactive endpoint testing.

Debugging: `Hooks.onOperatorDebug()` (dev only) adds assembly-time stack traces. `.checkpoint("description")` creates named reference points in error traces.

### Q12: "Why GraphQL over REST? What new problems does it introduce?"

**Why:** A pharmacy dashboard needs data from Order, Patient, Inventory, and Shipping services. With REST: 4 round trips or a BFF with a new endpoint per view. GraphQL: one query, frontend declares exactly what it needs.

**Problems:**
- **N+1 queries:** 50 orders with patients → 50 Patient service calls. Fix: DataLoader batches into one call.
- **Query complexity attacks:** Deep nesting can cripple the backend. Fix: depth limiting, complexity scoring, persisted queries.
- **Caching is harder:** REST caches by URL; GraphQL POSTs defeat HTTP caching. Fix: server-level and Redis-based caching.
- **Monitoring:** Every request hits `/graphql`. Fix: custom instrumentation tagged by operation name.

### Q13: "How do you prevent N+1 queries with DataLoader?"

**Without DataLoader:** resolving `order.patient` for 50 orders → 50 calls to `patientService.getById(id)`.

**With DataLoader:** All 50 patient IDs collected in the same execution tick → one call to `patientService.getByIds([...])`. Results mapped back.

Create DataLoader **per request** (not global — avoids stale data and permission leaks). In Spring for GraphQL:

```java
registry.forTypePair(String.class, Patient.class)
    .registerMappedBatchLoader((ids, env) ->
        patientService.findAllByIds(ids));
```

DataLoader handles batching, deduplication (same patient from two orders → fetched once), and per-request caching.

### Q14: "How do you handle field-level authorization in GraphQL?"

Two layers:

**1. Resolver-level:** `@PreAuthorize("hasRole('PHARMACIST')")` on the `patient.medications` resolver. Integrates with Spring Security and JWT claims.

**2. Schema directive:** Custom `@auth(requires: ROLE)` directive. Declarative, visible in the schema.

For PHI fields: **field-level masking**. If the user lacks PHI access, the resolver returns `"***-**-1234"` instead of null — UI renders gracefully, audit log records the attempt.

Authorization lives in the backend services too — if someone bypasses GraphQL and calls the Patient service directly, the same checks apply.

### Q15: "How would you design a GraphQL schema that federates data from multiple Spring Boot services?"

**Apollo Federation** via Netflix DGS Framework. Each service owns a schema slice; a gateway composes them.

Order service subgraph:
```graphql
type Order @key(fields: "id") {
  id: ID!
  status: OrderStatus!
}
```

Patient service extends:
```graphql
extend type Order @key(fields: "id") {
  id: ID! @external
  patient: Patient!
}
```

The `@key` directive tells the gateway how to correlate entities across services.

**Key decisions:**
- Each type has exactly one owning service
- Gateway caches query plans — computed once, reused
- If Shipping is down, queries not touching shipping fields still succeed (partial data with errors)
- Schema registry with CI/CD compatibility validation — breaking changes fail the build

### Q16: "How do you structure a Next.js application consuming a GraphQL API? Server vs. client components?"

Split by **interactivity**:

**Server Components** (default): page layouts, data tables, order summaries. Fetch via server-side `fetch` to GraphQL. HTML ships pre-rendered, no JS bundle. Fast first paint.

**Client Components** (`'use client'`): search filters, real-time status badges, modals, forms. Use Apollo Client's `useQuery`/`useMutation`. Apollo initialized once in a provider wrapping the layout.

State management: Apollo cache for server state. `useState`/`useContext` for UI state. Avoid Redux unless genuinely needed.

### Q17: "How do you handle real-time updates on the frontend — live order status tracking?"

Three options:

**GraphQL Subscriptions (WebSocket):** Best for high-frequency updates. Subscription resolver returns `Flux<OrderStatusEvent>` backed by a reactive Service Bus consumer. Apollo's `useSubscription` on the frontend. Downside: stateful connections complicate load balancing.

**Polling (`pollInterval: 5000`):** Simplest. 5-second delay is fine for most pharmacy dashboard use cases. Works behind any load balancer.

**Server-Sent Events:** Middle ground — server pushes over long-lived HTTP. Simpler than WebSocket but less flexible.

**At Empower:** Polling for the main dashboard (order list). WebSocket subscription for single-order detail view where a pharmacist actively watches compounding progress.

### Q18: "What's your approach to state management in a complex React application?"

Three categories, right tool for each:

**1. Server state:** Apollo Client's normalized cache. Handles fetching, caching, invalidation, optimistic updates. Cache keyed by `__typename:id` — mutations auto-update all components displaying that entity.

**2. Local UI state:** `useState` in the owning component. No need to globalize "is this modal open."

**3. Shared UI state:** React Context + `useReducer` scoped to the relevant subtree. Example: `DashboardFilterContext` holding active filters.

**Zustand:** Only if complex cross-cutting state makes Context re-renders expensive (e.g., drag-and-drop workflow builder). Not preemptively.

**Avoid:** Redux for new projects. Boilerplate isn't justified when Apollo + Context covers it.

### Q19: "How do you design an AKS cluster for a healthcare application?"

**Namespaces:** Per-environment minimum. Within prod, separate by domain: `pharmacy-orders`, `manufacturing`, `platform-services`. Namespace-level resource quotas and RBAC.

**Network policies:** Default-deny all ingress/egress. Explicitly allow only valid traffic paths. mTLS via service mesh for east-west traffic.

**Resource limits:** Explicit CPU/memory requests (p50 usage) and limits (~2x). HPA on custom metrics (queue depth, latency), not just CPU.

**Node pools:** Separate pools by workload — "general" for stateless APIs, "compute" for data-heavy operations. Confidential Computing nodes for PHI.

### Q20: "Service Bus topics vs. queues? Ordering and exactly-once?"

**Queues** for point-to-point (one producer, one consumer group). **Topics** for pub/sub (one event, multiple consumers).

**Ordering:** Service Bus **sessions**. Assign a session ID (e.g., order ID). FIFO guaranteed within a session.

**Exactly-once:** Aim for **effectively-once** via idempotency. Unique message IDs + deduplication store (Redis). Service Bus's built-in duplicate detection as a second layer.

---

### Q21: "Zero-downtime deployments on AKS? Database migrations during rolling deploys?"

**Default:** Rolling deployment with readiness probes. `maxUnavailable: 0`, `maxSurge: 1`.

**Critical services:** Canary via Istio/Flagger. 5% → 25% → 50% → 100%. Auto-rollback if error rate > 1%.

**Blue/green:** For infrastructure-level changes (K8s upgrades, node pool changes).

**Database migrations — expand-and-contract:**
1. **Expand:** Add new column, don't modify existing. Old code ignores it; new code writes to both. Runs before app deploy.
2. **Migrate:** Background job backfills new column.
3. **Contract:** Next release drops old column (all pods on new version).

Flyway/Liquibase in CI/CD. **Critical rule:** Never deploy a migration and the app change that requires it in the same release.

### Q22: "Walk me through your observability setup on AKS."

Three pillars, unified by correlation:

**Tracing (OpenTelemetry):** Trace ID propagated via W3C Trace Context. OpenTelemetry Java agent auto-instruments HTTP, DB, Service Bus. Exports to Azure Monitor Application Insights.

**Logging:** Structured JSON with trace ID, span ID, service name, operation. Ships to Azure Log Analytics. PHI operations: log action and user, never PHI data.

**Metrics (Prometheus + Grafana):** RED method + business metrics (orders/min, queue depth). Alerts: p99 > 500ms for 5 min, error rate > 1% for 3 min, dead-letter depth > 10.

Principle: alert → Grafana → traces → logs. Three clicks to root cause.

## III. AI/Data & Python (Q23–Q26)

### Q23: "How would you integrate a Python ML model into a Java/Spring Boot backend?"

Decouple via a **model-serving microservice**. Python model behind FastAPI (or TorchServe) on AKS. Spring Boot calls via async WebClient with Resilience4j circuit breakers.

Why not embed in Java? (1) Data scientists work in Python. (2) Model has its own deploy lifecycle — retrain weekly without touching order processing.

Contract: OpenAPI or gRPC. Model accepts features, returns prediction + confidence. Spring Boot decides what to do with low-confidence results (route to human review).

Batch inference: Python pipeline via Azure Data Factory or AKS CronJob.

---

### Q24: "How do you serve a model with sub-100ms latency at scale?"

Four levers:
- **Model optimization:** Quantize (FP32 → INT8) via ONNX Runtime. 2–4x faster.
- **Infrastructure:** No cold starts. Min 2 warm replicas. HPA on request latency.
- **Caching:** Drug interaction for "Lisinopril + Metformin" → same result every time. Redis cache by input hash. <1ms hit.
- **Async when possible:** "Checking interactions..." UI → submit via Service Bus, push result via WebSocket.

### Q25: "Concrete GenAI use case for pharmacy operations — production-ready, not a demo."

**Prescription intake triage.** Compounding pharmacy prescriptions arrive as faxed PDFs, e-prescriptions, phone-in notes. Pharmacist manually extracts data today.

GenAI: multimodal LLM extracts structured data → validation layer (rules engine checks NPI, formulary, dosage ranges) → pharmacist reviews pre-filled form.

Production-grade requirements:
- **Confidence scoring:** Per-field. Low-confidence highlighted for review. Pharmacist always has final approval.
- **Audit trail:** Input doc, model output, confidence, pharmacist edits all logged. FDA compliance.
- **Feedback loop:** Corrections feed fine-tuning.
- **Cost management:** Batch non-urgent. Small models for simple e-prescriptions. Large model for handwritten faxes.

### Q26: "How do you build data pipelines for both operational reporting and ML training?"

**Lambda-style** with shared event backbone:

**Real-time:** Domain events → Service Bus → stream processor → dashboards (Azure Data Explorer).

**Batch:** Same events → Data Lake (Parquet) → nightly Data Factory pipeline → training datasets (Databricks).

Key: **events are the single source.** Both paths consume the same stream.

**Feature Store** (Feast) prevents training-serving skew — ensures training features match production inference features exactly.

### Q28: "How do you establish engineering standards without becoming a bottleneck?"

**1. Paved roads:** Starter templates with OpenTelemetry, CI/CD, Dockerfile, K8s manifests. `create-service --name my-service` → production-ready in 5 min.

**2. ADRs:** Every decision documented. Scales influence without requiring presence.

### Q29: "Describe a technical trade-off between speed and sustainability."

**Situation:** Hard deadline for patient portal feature. Fastest path: add to existing monolith with heavy tech debt.

**Action:** Built as a **separate module within the same codebase** — own package structure, own schema, clean internal API. Speed (no new pipeline) + extractability (documented extraction plan).

**Result:** Shipped on time. Module extracted in 3 days (vs. 3 weeks for others). Team adopted "modular monolith" approach.

### Q30: "Walk me through your most complex production incident."

**Incident:** Intermittent 30-second latency spikes every 15–20 min. No errors, requests just hung. ~10% of users hit timeouts.

**Diagnosis:** Traced to database connection pool exhaustion. Queries were fast but connections leaked. Root cause: `subscribe()` in a reactive pipeline without error handling — upstream timeout cancellation didn't return connections to pool. Pool's idle timeout caused periodic "recovery."

**Fix:** Wrapped DB calls in `usingWhen` (guaranteed release even on cancellation). Added pool utilization metric with 80% alert. Custom lint rule for bare `subscribe()` calls.

### Q31: "How do you design for graceful degradation in a pharmacy context?"

Classify by **criticality:**
- **Critical:** Drug interaction alerts, dispensing verification. Synchronous + retries + local fallback caches.
- **Important:** Inventory counts, shipping estimates. Show cached data with "may be delayed" indicator.
- **Nice-to-have:** Recommendations, reports. Feature-flag off during incidents.

Patterns: circuit breakers (Resilience4j), bulkheads, local caches for critical reference data, queue-based leveling.

Every response includes a **degradation indicator** so the frontend communicates data freshness to users.

### Q32: "How do you design audit logging for PHI?"

Append-only store, separate from app databases. Every PHI access: `{timestamp, userId, action, resourceType, resourceId, fieldsAccessed, ipAddress}`. Immutable.

`@AuditPHI` annotation on service-layer methods. AOP interceptor publishes to Service Bus → Audit Service persists.

**Never log PHI data itself.** Log "User X accessed Patient Y's medication list" — not the medications.

6-year retention on immutable blob storage. Access to audit logs is itself audited. Anomaly detection for unusual access patterns.

### Q33: "What architectural patterns ensure data integrity and traceability in regulated environments?"

- **Event sourcing:** Every state change is an immutable event. Current state = replayed events. Tamper-evident history.
- **CQRS:** Separate write (append events) from read (materialized views). Add regulatory views without changing writes.
- **Immutable infrastructure:** Every deploy = new container image with SHA tag. Regulator asks "what ran on March 15?" → exact image hash.
- **Cryptographic integrity:** Hash chains on audit events. Tampering breaks the chain. DEA expects this for controlled substances.

**End-to-end:** Prescription modification → validate → append immutable event → update read model → audit log → hash chain. Every step independently verifiable.
