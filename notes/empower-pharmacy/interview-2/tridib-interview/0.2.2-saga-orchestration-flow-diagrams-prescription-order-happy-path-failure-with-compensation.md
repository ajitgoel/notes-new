## 1. Physical Architecture — Services, Queues, and Connections

```
SERVICES (Spring Boot apps on AKS)          QUEUES (Azure Service Bus)

┌──────────────────────────┐
│   OrchestratorService    │
│                          │       ┌─────────────────────────┐
│  Sends commands to ──────┼──────▶│  validation-commands     │◀── ValidationService listens
│                          │       ├─────────────────────────┤
│                    ──────┼──────▶│  inventory-commands      │◀── InventoryService listens
│                          │       ├─────────────────────────┤
│                    ──────┼──────▶│  payment-commands        │◀── PaymentService listens
│                          │       ├─────────────────────────┤
│                    ──────┼──────▶│  compounding-commands    │◀── CompoundingService listens
│                          │       ├─────────────────────────┤
│                    ──────┼──────▶│  shipping-commands       │◀── ShippingService listens
│                          │       └─────────────────────────┘
│                          │
│                          │       ┌─────────────────────────┐
│  Reads replies from ◀────┼──────│  saga-replies            │◀── ALL services reply here
│                          │       └─────────────────────────┘
└──────────────────────────┘
          │
          ▼
   ┌──────────────┐
   │  saga_db     │  ← Persists saga state (sagaId, step, results)
   │  (Postgres)  │
   └──────────────┘
```

## 4. When a Compensation Itself Fails — Retry + Dead-Letter

```
OrchestratorService            Queue                      Participant Service
       │                              │                              │
  Compensating Step 2: Release Inventory
       │                              │                              │
       │  ReleaseReservation          │                              │
       ├─────────────────────────▶ inventory-commands ─────────▶ InventoryService
       │                              │                          SERVICE IS DOWN!
       │                              │                              │
       │  No reply within 30s...      │                              │
       │                              │                              │
       │  RETRY 1 (after 1s)          │                              │
       ├─────────────────────────▶ inventory-commands          still down
       │                              │                              │
       │  RETRY 2 (after 5s)          │                              │
       ├─────────────────────────▶ inventory-commands          still down
       │                              │                              │
       │  RETRY 3 (after 30s)         │                              │
       ├─────────────────────────▶ inventory-commands          still down
       │                              │                              │
       │  All retries exhausted        │                              │
       │                              │                              │
       ├─────────────────────────▶ inventory-commands-dlq     ← Dead-Letter Queue
       │                              │
  saga status = COMPENSATION_IN_PROGRESS
  ALERT → PagerDuty → Ops engineer investigates
       │                              │
  Later: Ops fixes InventoryService, replays message from DLQ
       │                              │                              │
       │                         inventory-commands ─────────▶ InventoryService
       │                              │                    idempotency check → processes
       │         {status: SUCCESS}    │                              │
       ◀────────────────────────── saga-replies ◀─────────────────  │
       │                              │                              │
  saga status = COMPENSATED           │                              │
```

## 5. Key Takeaway — The Communication Pattern

```
Every interaction follows this pattern:

Orchestrator ───── command message ─────▶ [service]-commands queue ─────▶ [Service]
                                                                            │
                                                                      does work
                                                                      commits to own DB
                                                                            │
Orchestrator ◀──── reply message ────────  saga-replies queue  ◀───────────  │
     │
updates saga_db
decides next step
(advance or compensate)

No HTTP. No direct service-to-service calls. Only messages on queues.
Each service is its own Spring Boot app, its own DB, its own AKS deployment.
The queues are the only coupling between them.
```
